# WASIX Dynamic Extensions

This document records the production design for SQL extensions in
`pglite-oxide`. The active path is PGlite built as a WASIX dynamic main module,
with matching extension side modules loaded by Wasmer/WASIX.

Extension productization is Phase 4 of
[WASIX_WASMER_ROADMAP.md](WASIX_WASMER_ROADMAP.md). The roadmap also defines the
earlier source/build and correctness gates extensions depend on.

## Goal

Support PGlite SQL extensions with Rust-native startup and no JavaScript loader:

- PGlite main module at `/bin/pglite`
- Postgres data directory at `/base`
- extension libraries under `/lib/postgresql`
- extension SQL/control files under `/share/postgresql/extension`
- `pg_dump` built from the same pinned source tree
- Wasmer LLVM AOT artifacts generated by CI and loaded by headless Wasmer

The runtime must install only the extensions requested by the developer while
allowing asset packs to contain the full supported catalog.

## Proven Protocol

The spike established the bridge Rust needs to drive the backend:

- WASIX args and env seed a normal PGlite process layout
- host filesystem is mounted at `/`
- `my_exec_path` is seeded to `/bin/pglite`
- Rust calls `_start` once, then uses the PGlite lifecycle exports
- Rust sends a Postgres startup packet with `ProcessStartupPacket`
- `pgl_wasix_input_*` and `pgl_wasix_output_*` move frontend/backend protocol
  bytes across the host/runtime boundary
- `PostgresMainLoopOnce`, `PostgresSendReadyForQueryIfNecessary`, and
  `pgl_pq_flush` drive each protocol message

This is the protocol implemented in `src/pglite/postgres_mod.rs`. The
JavaScript/Emscripten-era `pgl_startPGlite` and `pgl_setPGliteActive` entrypoints
are intentionally not part of the Rust WASIX ABI.

## Upstream PGlite Foundation

Use `postgres-pglite` branch `REL_17_5_WASM-pglite-builder` as the upstream
Postgres/PGlite foundation. This branch is the current source of truth for the
PGlite-specific Postgres shape we should build on:

- backend lifecycle exports such as `pgl_initdb`, `pgl_backend`, and
  `pgl_shutdown`
- initdb, boot, and single-user backend handling
- protocol-loop structure through `interactive_one.c` and `pg_proto.c`
- extension source catalog and packaging layout
- extension import/export discovery
- exported hook coverage for extensions such as AGE and vector

Our production work should layer the WASIX/Wasmer backend onto that foundation
rather than carrying a separate PGlite C runtime fork. The upstream branch gives
us the PGlite semantics and extension catalog; `pglite-oxide` supplies the
production Rust-hosted WASIX execution model, AOT packaging, cache layout, and
CI-owned asset generation.

## Build Requirements

The maintained source patch should become a `wasix-dl` build personality on top
of `REL_17_5_WASM-pglite-builder`:

1. Compile Postgres/PGlite with WASIX, exceptions, and PIC.
2. Build the main module as a dynamic executable with a stable export list.
3. Build extension side modules from the same sysroot and source tree.
4. Preserve canonical Postgres install paths.
5. Generate extension archives with deterministic ordering, mtimes, owners, and
   modes.
6. Generate `pg_dump` from the same configured tree.
7. Record every source pin, toolchain version, archive hash, import/export set,
   and smoke-test result in the manifest.

Extension artifacts from different runtime builds must not be mixed.

## Builder Branch Reuse

The builder branch contributes several pieces we should reuse directly or port:

- `.buildconfig` pins PostgreSQL 17.5, `REL_17_5_WASM`, and SDK version
  `3.1.74.11.11`.
- `wasm-build/build-pgcore.sh` separates core build, install, link, symbol
  extraction, and artifact distribution.
- `wasm-build/build-ext.sh` iterates contrib extensions, packages successful
  builds, and runs the extra extension scripts.
- `wasm-build/getsyms.py`, `linkimports.sh`, and `linkexport.sh` define the
  export/import discovery loop required by dynamically loaded extensions.
- `pglite/Makefile` is the current upstream catalog for external SQL
  extensions: `pg_ivm`, `vector`, `pgtap`, `pg_uuidv7`, `age`, and
  `pg_hashids`.
- `pglite-wasm/excluded.pglite.imports` tracks `_invoke_*` symbols needed
  by AGE-style function-pointer patterns.

The pinned `pglite-build` checkout is useful as historical/build evidence, but
the current `REL_17_5_WASM-pglite-builder` branch already carries the shared
`wasm-build` scripts we need. `xtask assets source-spine` checks that the
important scripts still match `pglite-build` so the production build has one
source root: the configured `postgres-pglite` tree.

Important caveats from the branch audit:

- its current `WASI=true` path skips contrib extension builds; the production
  path still needs a new `wasix-dl` personality rather than a flag flip
- `.buildconfig` uses host `zic` by default, so timezone/tzdata inputs must be
  pinned or packaged from a reproducible build image
- `pglite/Makefile` stages both `share/extension` and
  `share/postgresql/extension`; the current asset packaging normalizes runtime
  and extension archives to the canonical PostgreSQL layout
- upstream `pack_extension.py` writes non-deterministic `.tar` files and mutates
  `PGROOT` after packaging; keep its install-delta and import-extraction model,
  but implement release archives in `xtask` as deterministic `.tar.zst`
- the branch's JavaScript-facing C API is `pgl_initdb`, `pgl_backend`, and
  `interactive_one`; the Rust protocol bridge exports in
  `src/pglite/postgres_mod.rs` must be a deliberate maintained ABI layer on top,
  not an accidental fork
- the Rust host enforces that layering in source: lifecycle exports are loaded
  separately from WASIX protocol exports, and `xtask assets check` rejects a
  regression back to a generic export bucket or JS/Emscripten lifecycle probes
- newer PGlite fixes landed outside this branch, including checkpointer disable,
  background-worker disable, default `postgres` identity, imported memory sizing,
  and artifact caching; source sync must audit those before every asset release

The layering boundary is:

- keep the branch's PGlite lifecycle, protocol loop, extension catalog, and
  symbol/export machinery as upstream-aligned input
- implement the WASIX dynamic-linking build personality in our maintained patch
- build main and side modules from the same configured tree
- expose Rust extension constants only after smoke tests pass against the
  WASIX/Wasmer runtime

## Extension Catalog

The generated catalog should merge:

- PGlite's extension catalog
- PGlite REPL exports
- `postgres-pglite/pglite/other_extensions`
- supported Postgres contrib directories
- pinned external extension repositories

Public constants are emitted only after a Rust smoke test passes. `vector` is
the initial required proof because it exercises side-module loading, type I/O,
operators, indexes later, and server-mode clients.

The manifest must also classify extension startup requirements:

- normal `CREATE EXTENSION` after files are installed
- preload-required extensions such as `pg_stat_statements`
- startup-config extensions that require postmaster-level GUCs or symbol exports,
  such as `pgoutput` / logical-decoding paths
- extensions with native library dependency order requirements, such as PostGIS
- PGlite JavaScript plugins, such as `live`, which should not be exposed as SQL
  extension constants until a Rust-native API exists

## Runtime Installation

The runtime cache stores immutable extracted trees by manifest hash. Extension
caches store immutable extracted extension trees by archive hash. Opening a
database hardlinks runtime and requested extension files into the per-instance
root, falling back to copies if hardlinks fail.

`CREATE EXTENSION IF NOT EXISTS <sql_name>` is run only after the extension
files are present in the instance root. Archive extraction must reject absolute
paths, parent traversal, unsafe symlinks, and non-canonical Postgres locations.

## AOT Loading

CI produces target-specific Wasmer LLVM artifacts for every supported host. The
runtime verifies the raw artifact SHA before unsafe deserialization. Compressed
artifacts may be used to satisfy package-size limits, but they are expanded once
into the persistent cache and loaded raw afterward.

Unsupported targets fail early with an actionable error. Local LLVM compilation
belongs in `xtask` and CI only.

## Acceptance

Required tests before promoting an extension constant:

- direct `Pglite` can `CREATE EXTENSION`
- direct `Pglite` can use at least one representative function/operator/type
- `PgliteServer` plus a normal Postgres client can use the extension
- archive extraction rejects unsafe paths
- manifest SHA validation passes
- AOT SHA verification and deserialization pass
- preload/startup-config extensions prove their config was applied before
  backend start
- macOS arm64/x64 pass multi-module exception/error-recovery tests for extension
  side modules

Required tests before exposing `pg_dump`:

- dump a database through the WASIX `pg_dump` module
- restore the SQL into a fresh database
- verify restored schema and rows
- verify extension OIDs are dumpable and dumps include `CREATE EXTENSION` for
  extensions such as `vector`
- start with plain SQL and `--inserts`; add COPY restore support only after the
  protocol path proves it can preserve synchronization
